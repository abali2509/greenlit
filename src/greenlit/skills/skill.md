# Reading a greenlit prompt

A greenlit prompt is a structured task specification with up to nine sections. Each section answers a specific question and must be interpreted on its own terms. Prompt files are typically found under `.greenlit/` in one of three formats:

- **XML** — wrapped in `<prompt type="...">` with `<section key="...">` children
- **Markdown** — `# PROMPT` header, one `## SECTION` per section
- **JSON** — `{ "task_type": "...", "sections": { "ask": "...", ... } }`

All three are semantically equivalent. The section keys are always: `ask`, `goal`, `context`, `scope`, `delegation`, `inputs`, `outputs`, `constraint`, `attention`.

## Task types

Each prompt declares a task type. The task type shapes what the user expects back:

| Type | Expectation |
|------|-------------|
| `review` | Code/PR/architecture review — return findings and recommendations, **not** code changes |
| `plan` | Architecture, design, task breakdown — return a plan for approval, **not** implementation |
| `action` | Implementation, refactoring, migration — write code |
| `debug` | Diagnose failures, trace bugs — report root cause, then fix |
| `research` | Spikes, investigations, trade-offs — gather information and summarise |

Custom task types can extend one of the above via YAML templates — treat them as a specialised variant of their base type.

## Section semantics

Read every section. Treat them as a layered contract, not a checklist.

1. **ASK** — The literal request. Restate it in one line before acting.
2. **GOAL** — The *why*. Use this to disambiguate the ASK when multiple interpretations exist. If your plan doesn't move toward the GOAL, you're off track.
3. **CONTEXT** — Background facts the user already knows. Don't re-derive them; use them to frame your thinking.
4. **SCOPE** — Hard IN / OUT / DEFERRED boundaries. Anything in OUT is off-limits even if it looks helpful. DEFERRED is "not now."
5. **DELEGATION** — Subagent roles. If present, use these to parallelise work or protect the main context.
6. **INPUTS** — Files, paths, URLs, and data the user has supplied. Read these before doing anything else.
7. **OUTPUTS** — The expected deliverables. Shape your response to match this exactly — format, structure, granularity.
8. **CONSTRAINT** — Non-negotiable rules. Violating a CONSTRAINT is a failure even if the OUTPUT is otherwise correct.
9. **ATTENTION** — Red flags, edge cases, preconditions. Often contains instructions like "restate the ask first" or "ask clarifying questions before starting." Check this **first** so you don't skip a precondition.

Missing sections mean "no constraint / no preference" — don't invent content for them.

## Execution protocol

When handed a greenlit prompt:

1. **Read ATTENTION first.** It often contains preconditions that gate the rest of the work.
2. **Parse all present sections** into a mental model. Note which sections are empty.
3. **Restate the ASK** in one line so the user can confirm you understood.
4. **Ask clarifying questions** if ATTENTION requests it, or if ASK and CONSTRAINT conflict, or if SCOPE is ambiguous. Do not guess.
5. **Verify INPUTS are accessible.** For each artifact listed in INPUTS (files, paths, URLs, data), confirm you can read or access it before proceeding. If any are missing or inaccessible, surface this to the user immediately — do not continue with incomplete inputs.
6. **Honor SCOPE.** Never touch OUT. Note DEFERRED items and move past them.
7. **Apply task-type defaults.** `plan` and `review` return deliverables without writing code. `action` writes code. `debug` reports findings and fixes. `research` summarises.
8. **Match OUTPUTS exactly.** If the prompt asks for a bulleted task list with one-line commit messages, return exactly that — no prose preamble, no extra sections.
9. **Verify against CONSTRAINT** before responding.

## Common failure modes

- **Treating the prompt as a suggestion.** Every section is load-bearing. Skipping CONSTRAINT or ATTENTION is the most common cause of rework.
- **Scope creep.** If the user wrote `OUT: tooling changes`, don't suggest tooling changes — not even as a footnote.
- **Ignoring the task type.** A `plan` prompt does not want code. An `action` prompt does not want a plan — it wants the change.
- **Skipping the restate step** when ATTENTION asks for it. It's a cheap alignment check and the user relies on it.
- **Inventing content for empty sections.** If DELEGATION is blank, the user does not want subagents. If CONTEXT is blank, ask — don't fabricate background.
